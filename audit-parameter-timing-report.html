The headline: a parameter's “calculation time” is mostly the wrong question

Auditify loads each page once, then ~80% of the parameters are extracted or computed from that already-loaded data — a regex over the HTML, one page.evaluate(), or a field read out of a cached API response. Those cost 0.1–80 ms each. They feel “free” because they ride on a handful of shared cost drivers that do the real waiting.

Take the two you asked about: TBT is just audits["total-blocking-time"] read out of a PageSpeed response — its own work is <1 ms. But that response is a ~50 s PageSpeed API call (measured: 50 s mobile, 75 s desktop) shared with LCP, CLS, FCP, TTFB, INP, FID, SI, the PageSpeed score and 4 more. Inventory page speed is the opposite: it opens its own browser tab and navigates to the inventory page, so it owns a real ~2–15 s cost nobody else shares.

So this report is organized around cost drivers, not a flat per-row stopwatch. Each of the 194 parameters below shows both its own marginal compute and the driver it depends on.

EHow to read these numbers

Measured vs. modeled
  • Measured (green badges): captured on a live run against a real dealership (galpinford.com, mobile) via the project’s own Puppeteer_Cheerio loader + the real metric services — Backend/_timing_calibration.mjs. Directly timed: PageSpeed mobile 50.4 s / desktop 75.1 s; full Technical category 42.8 s; page-setup 2.24 s; axe-core / whole Accessibility category 0.25 s; On-Page SEO incl. link checks 6.67 s; AEO/GEO incl. crawl+KG 4.0 s; inventory-path probe 2.25 s; and the combined UX 20 ms + Conversion 30 ms + AIO 10 ms.
  • Modeled (amber badges): derived from the operation each parameter performs in the code (request counts × typical round-trip). The entire Security & Compliance category is modeled, not run live, because it performs active SQLi/XSS injection, admin-path brute-forcing and credential testing — that is attack traffic and must not be fired at a third-party site you don’t own.
Why per-parameter times look tiny
  • The page is fetched once (browser launch + render + bot handling). After that, a cheerio parse ($) or a single page.evaluate() is pure CPU on data already in memory — microseconds to tens of milliseconds.
  • All Core Web Vitals (LCP, CLS, FCP, TTFB, INP, FID, TBT, SI) + PageSpeed score + Rendering/Lazy/Third-party/JS-execution are field reads out of one cached PageSpeed/Lighthouse response. None of them “costs” more than a field lookup.
  • All ~18 axe-derived accessibility checks are reads out of one axe-core run.
  • The AEO “Gemini / ChatGPT / Perplexity platform scores” are a deterministic weighted matrix over already-computed signals — no LLM is called, so they’re ~0 ms despite the names.
Parallelism & the single-instance caveat
  • The 7 categories run concurrently; inside Technical, the 2 PageSpeed calls + 3 page navigations run in one Promise.all. So total audit time ≈ the slowest single driver, not the sum of the rows below.
  • Adding up the “Effective time” column therefore overcounts — shared drivers are paid once. Use §B for true wall-clock.
  • Single-audit metrics are buffered in process memory and batch-written to Mongo, so the design assumes a single app instance (per the existing in-memory batching note).
